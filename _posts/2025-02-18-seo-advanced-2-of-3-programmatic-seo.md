> 关键词提示：程序化 SEO、自动化 SEO、SEO 工作流、GitHub Actions、静态博客 SEO。

大多数人做 SEO 是「运营思维」：写文章、改标题、发链接。作为开发者，你多了一条路：

> 把 SEO 当作一个「可以工程化和自动化」的系统，把重复工作交给脚本和 CI。

这一篇 SEO 高级实战 2/3，我们就用你现在的静态博客（Jekyll + GitHub Pages）为例，聊聊**程序化 SEO / 自动化 SEO** 的几个实用场景。

## 一、什么是「程序化 SEO」？

简单理解：

- 不是用脚本批量制造垃圾页面；
- 而是用代码和模板，**系统化管理和放大高质量内容**。

几个关键要素：

- 模板化：一类页面有固定结构（比如系列导航、索引页）；
- 批量化：结构相同，只是填入不同数据（文章列表、标签列表等）；
- 自动检查：在提交或构建阶段自动检查常见 SEO 问题（缺 title、缺 excerpt 等）。

对于你的技术博客来说，现在已经有了一些「程序化 SEO 的雏形」：

- 基于 `series` 的通用系列导航组件；
- `/seo/` 分类页按 `categories: seo` 自动列出所有 SEO 文章；
- 首页根据 `series` 推荐入门、进阶系列。

我们可以在此基础上，把自动化程度再往前推几步。

## 二、静态博客中适合程序化处理的 SEO 任务

### 1. 系列导航与索引页的自动生成

你现在已经有：

- `_includes/series-nav.html`：根据 `series` 自动生成系列导航；
- `seo.md`：按 `categories: seo` 自动展示 SEO 文章列表。

可以继续扩展：

- 为未来的技术系列提前约定：
  - 如 `series: python-beginner`、`series: go-concurrency`；
  - 首页自动加一个「该系列推荐」区块；
- 为各大主题自动生成索引页，例如：
  - `/python/`：全部 Python 文章 + 系列导航；
  - `/ai/`：LLM / AI 工具相关内容索引。

只要使用 Liquid 循环（`site.posts | where: 'series', 'xxx'`），就可以非常容易地实现「程序化索引」。

### 2. 自动检查 front matter 中的 SEO 关键信息

一个很实用的程序化 SEO 思路是：

> 用脚本对 `_posts` 做「体检」，找出缺少必要字段的文章。

例如，你可以写一个简单的 Python 或 Go 脚本：

- 扫描 `_posts/*.md`；
- 解析 YAML front matter；
- 检查每篇文章是否有：
  - `title`
  - `categories`
  - `tags`
  - `excerpt`
- 输出一个表格/列表：哪些文章缺了什么。

这样，你就可以集中精力修复这些「SEO 基础不完整」的文章，而不用一篇一篇人工翻。

### 3. 自动生成某些类型的内容块

例如：

- 在每篇 SEO 系列文章底部自动添加：
  - 指向 `/seo/` 分类页的链接；
  - 指向入门/进阶/高级系列的导航；
- 在代码类文章中自动检测是否包含代码块，如果没有，就提醒自己「这篇技术文缺少示例」。

这些都可以通过模板 + 少量脚本来实现。

## 三、在 GitHub Actions 中加入 SEO 质量门槛

既然你的博客已经托管在 GitHub 上，可以顺手把 SEO 检查集成到 CI 流程里。

思路：

1. 添加一个简单的检查脚本，例如 `scripts/seo_lint.py`；
2. 在 GitHub Actions 工作流中：
   - 在构建前运行这个脚本；
   - 如果发现严重问题（比如新文章缺 `title` 或 `categories`），CI 失败；
   - 把错误信息打印出来，提示你在提交前修正。

伪代码（工作流 YAML 片段）示例：

```yaml
- name: Run SEO lint
  run: |
    python scripts/seo_lint.py
```

这样每次 push 新文章时：

- CI 会帮你「守门」，避免把严重不合格的内容推到线上；
- SEO 规范变成自动检查的「项目规则」，而不是记忆里的 checklist。

## 四、程序化 SEO 不是「批量造站」，而是「系统性放大好内容」

很多人一听到「程序化 SEO」，容易联想到：

- 批量生成低质量 landing page；
- 采集 + 拼接内容。

这里要特别强调：

> 对个人技术博客来说，程序化 SEO 的核心目标，是 **节省脑力，把精力集中在内容本身**。

你可以交给代码/模版的事情包括：

- 内容的结构化展示（系列、索引、导航）；
- 元数据的完整性检查（title、excerpt、tags…）；
- 站点级的 SEO 规则统一（permalink、sitemap、robots 等）。

## 五、一个可执行的程序化 SEO 行动计划

结合你现在的博客状态，可以按下面步骤推进：

1. **确立系列约定**
   - 所有系列文章都用 `series` + `series_order`；
   - 首页为每个系列增加推荐区块；
   - 系列文章底部通过系列导航组件自动互链。

2. **写一个简单的 SEO Lint 脚本**
   - 检查所有 `_posts` 的 front matter；
   - 输出缺 `title` / `excerpt` / `tags` / `categories` 的文件列表；
   - 每月或每次大改前跑一遍，逐步补全。

3. **在 CI 中集成 Lint**
   - 对新 PR 或新 commit 启用 SEO Lint；
   - 让「缺少重要 SEO 字段」变成构建不通过的原因之一。

4. **逐步扩展索引页与主题页**
   - 先从 `/seo/`、`/python/` 等几个大类开始；
   - 以后新增主题线时（比如 AI 工具实战），顺手加对应索引页。

## 六、小结：开发者做 SEO 的优势，就是可以把它「工程化」

通过这篇《SEO 高级实战 2/3》，我们完成了从「文章级优化」到「系统级自动化」的一步迁移：

- 借助模板与脚本，自动生成系列导航和索引页；
- 用 Lint 脚本 + CI 把 SEO 基础规范化；
- 把重复而机械的 SEO 任务交给代码，让自己专注在真正有价值的内容创作上。
